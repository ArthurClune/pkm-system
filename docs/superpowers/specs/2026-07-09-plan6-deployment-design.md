# Plan 6: Deployment, backup & hardening — Design

Approved 2026-07-09. Implements Section 5 of
`2026-07-08-roam-migration-pkm-design.md`, updated by the decisions below.
Input context: `docs/superpowers/handoffs/2026-07-09-plan6-handover.md`.

## Goal

Run the PKM app as a permanent service on the Mac mini ("biber"), reachable
from the tailnet, with nightly backups into one sync-friendly directory, plus
the hardening/bugfix carry-forwards deferred to deployment.

## Decisions (from brainstorming)

- Deployment target is **this machine** (biber, macOS 26.5.2, Tailscale 1.98.5
  installed) — everything is installed and verified live during the plan.
- **Separate prod checkout**, not the dev repo: dev work can't break the
  running service, and a test server with its own scratch graph stays possible.
- Service home is **`~/.config/pkm/`** (app clone, live data, backups, logs).
- **Dual exposure**: Tailscale Serve HTTPS (browsers — secure cookies, Safari
  clipboard) *and* a direct bind on the machine's Tailscale IP
  (100.104.x.y:8974, plain HTTP for tailnet API clients). `cookie_secure`
  stays `true`.
- Backups are **local only**; off-machine sync is the user's job, so
  everything lands in one `backups/` dir they can point Syncthing/rsync at.
- **Git history for exported markdown only**; assets are mirrored beside it
  but gitignored (git-committing 2 GB of immutable content-hashed blobs
  doubles their footprint for no history value). This amends the original
  Section 5 wording "markdown + assets export into a git-committed directory".
- All deploy files live **in the repo as templates** with `{{PLACEHOLDER}}`
  markers — no secrets, usernames, or machine paths committed (repo may go
  public).
- Upload size cap default **150 MB**: the live graph already has 6 assets over
  25 MB, the largest a 96.7 MB PDF; 150 MB clears that with headroom and
  remains configurable.
- In-scope carry-forwards: asset upload hardening, `visibilitychange` draft
  flush, doomed `update_text` guard, `SyncContext` loud default.

## Section 1: Layout, templating, install

```
~/.config/pkm/
  app/          # dedicated clone of this repo (prod checkout)
  data/         # live graph: pkm.sqlite3, assets/, config.json
  backups/      # the one dir external sync points at (Section 4)
  logs/         # server + backup job stdout/stderr
```

New repo directory `deploy/`:

- `com.PLACEHOLDER.pkm.server.plist.template`,
  `com.PLACEHOLDER.pkm.backup.plist.template` — LaunchAgent definitions with
  `{{HOME}}`, `{{UV}}`, `{{LABEL_PREFIX}}`-style placeholders.
- `install.sh` — idempotent: fills templates (home dir, `uv` path, Tailscale
  IP via `tailscale ip -4`, label prefix), copies plists into
  `~/Library/LaunchAgents/`, `launchctl bootstrap`s them, configures
  `tailscale serve --bg`, creates the `~/.config/pkm` skeleton if missing.
  Re-running it is the upgrade path for service definitions.
- `update.sh` — run in the prod clone: `git pull`, `uv sync`,
  `pnpm install && pnpm build`, `launchctl kickstart -k` the service.
- `smoke.sh` — post-install verification (Section 6).
- `README.md` — what each piece does, install/update/restore-from-backup
  procedures.

Secrets (`password_salt`/`password_hash`, `session_secret`) exist only in
`~/.config/pkm/data/config.json`, generated by the existing
`pkm.server.setup` flow — never in the repo.

One-time migration: move the live graph from the dev repo's `data/` into
`~/.config/pkm/data/` with the dev server stopped, verified by pre/post
page/block counts.

## Section 2: Server bind changes

`run.py`/`config.py` gain a **list of bind hosts**: `bind_hosts` in
`config.json` (default `["127.0.0.1"]`), overridable by a repeatable `--host`
flag. Implemented with uvicorn's programmatic API and multiple pre-bound
sockets — one process listening on every configured host at the same port.

Production config: `["127.0.0.1", "<tailscale-ip>"]` (literal IP filled at
install; Tailscale node IPs are stable). Consequences:

- Local access and Tailscale Serve (which proxies to 127.0.0.1) work even if
  the tailnet is down.
- Tailnet clients can hit `http://100.104.x.y:8974` directly. Browsers use
  the Serve HTTPS URL; the direct HTTP port is for API scripts.
- If the Tailscale IP isn't bindable yet at boot, the server exits and
  launchd `KeepAlive` retries until Tailscale is up.

## Section 3: launchd service + Tailscale Serve

`com.<label>.pkm.server.plist` (LaunchAgent — runs in the user session where
the Tailscale app lives):

- `ProgramArguments`: `uv run --project <home>/.config/pkm/app/server python
  -m pkm.server.run --data-dir <home>/.config/pkm/data` (absolute paths,
  `WorkingDirectory` = app clone).
- `RunAtLoad` + `KeepAlive` (crash restart; also the boot-time bind retry),
  `ThrottleInterval` ~10 s.
- stdout/stderr → `~/.config/pkm/logs/server.{out,err}.log`.

Tailscale Serve: `tailscale serve --bg --https=443 http://127.0.0.1:8974` —
persists in Tailscale's own state across reboots; a one-time `install.sh`
step, not a launchd job. App URL: `https://biber.<tailnet>.ts.net`.

## Section 4: Nightly backup + export

Second LaunchAgent, `StartCalendarInterval` ~03:30, runs
`uv run python -m pkm.backup --data-dir … --backups-dir …`. All logic is a
tested module in the `pkm` package; the plist is a thin shell. launchd runs
missed calendar jobs on wake, so a sleeping mini doesn't skip a night.

```
backups/
  sqlite/pkm-YYYY-MM-DD.sqlite3   # online backup from a mode=ro source
                                  # (pattern proven in plan 5's smoke);
                                  # keep last 14 daily + last day of each month
  export/
    .git/                         # local-only repo, one auto-commit per run
    pages/<slug>.md               # full markdown export (new exporter)
    journal/YYYY-MM-DD.md
    assets/                       # incremental mirror of live assets (gitignored)
```

Exporter: markdown preserves outline structure (indented bullets), TODO
state, `[[refs]]`/`#tags` as written; block refs render as the target text
plus a marker; asset links rewrite to relative `assets/…` paths so the export
is self-contained and human-readable. Filenames are sanitized page titles
with collision handling. The asset mirror is content-hash-aware: only new
hashes copy in, deleted hashes prune — a no-change night costs seconds.

Failure behavior: a failing step leaves existing backups untouched, logs to
`logs/backup.log`, exits nonzero (visible via log + launchctl last-exit
status). No partial rotation: the new sqlite file writes to a temp name and
renames in before old files are pruned. Never opens the live DB for writing.

## Section 5: Hardening + carry-forward fixes

Asset upload hardening (`routes_assets.py`):

- Size cap: 413 above `max_upload_bytes` (config, default 150 MB).
- Upload mime allowlist (images, PDF, plain text, office docs) — 415 otherwise.
- Serving: `X-Content-Type-Options: nosniff` on all asset responses;
  `Content-Disposition: inline` only for an explicit safe list (raster
  images + PDF), `attachment` for everything else — **SVG is explicitly
  excluded from inline** (it can carry scripts), so uploaded HTML/SVG can
  never execute in the app's origin.

Frontend/client fixes (each small, each tested):

- `visibilitychange` flush of pending debounced text ops — closes the
  close-tab-within-500 ms data-loss window.
- Doomed `update_text` guard: `useOutline.takePendingTextOps` returns nothing
  for a block a remote batch just deleted, so one stale op can't reject a
  whole batch and force a desync refetch.
- `SyncContext` loud default: the no-provider `enqueue` throws instead of
  silently dropping writes.

## Section 6: Testing & verification

- **pytest**: bind-hosts config parsing; exporter golden files against
  fixture graphs with round-trip accounting (every page/block/asset appears);
  backup rotation (temp-then-rename, daily pruning, monthly retention);
  upload cap/allowlist/headers.
- **vitest**: the three frontend fixes.
- **`deploy/smoke.sh`** against the real running service: login over the
  Serve HTTPS URL; authenticated API read; a **real WebSocket connection**
  (the `websockets>=13` lesson from plan 5); asset request with and without
  cookie (200/401); direct-bind `http://100.104.x.y:8974` health check.
- **Backup verification**: one manual run; open the sqlite copy read-only and
  compare page/block counts to live; check the export git log; confirm a
  no-change re-run commits nothing.
- Live-data safety rules (handover) hold throughout: tooling never opens the
  live DB for writing; the `data/` migration happens with the dev server
  stopped and is verified by pre/post counts.

## Out of scope

- Off-machine backup sync (user-managed against `backups/`).
- CI, containers, LaunchDaemon/system-level service.
- Remaining plan-5 carry-forwards not listed above (click-to-focus caret
  mapping, backlink snapshot refresh, journal scroll reset, split-then-type
  remount race, `useAutocompleteField` extraction, e2e harness nits) — they
  stay on the spec's durable list.
